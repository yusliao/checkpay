## 模块职责

存放 CheckPay 收款系统的完整设计文档，包括技术栈架构决策、数据库设计和页面交互设计。所有设计文档同时提供 Word 和 Markdown 两种格式。

## 文档清单

### 1. 技术栈架构决策文档

**文件**: `收款系统_技术栈架构决策_V1.0.docx` / `.md`

**内容概要**:
- 三大核心决策：Blazor Server（前后端不分离）、Railway（云平台）、Azure Document Intelligence（OCR）
- 完整技术栈：.NET 10 + PostgreSQL + MudBlazor + EF Core + Entra ID
- Solution 项目结构（6个项目）
- Railway 部署指南
- OCR 异步处理架构
- 开发路线图（P0-P5）
- 月度成本估算（~$17/月）

**关键决策**:
- 选择 Blazor Server 而非 React：一套代码、零跨域、SignalR 内置
- 选择 Railway 而非 Azure App Service：节省 83% 成本
- 保留 Azure Document Intelligence：prebuilt-check 模型无可替代

### 2. 数据库设计文档

**文件**: `收款系统_数据库设计_V1.0.docx` / `.md`

**内容概要**:
- 设计原则：UTC 时区、UUID 主键、软删除、乐观并发、JSONB 扩展
- 核心表结构（6张表）：
  - `check_records` - 支票记录（9个字段 + 审计字段）
  - `debit_records` - 扣款记录（10个字段 + 审计字段）
  - `ocr_results` - OCR 原始结果（JSONB 存储）
  - `customers` - 客户主数据
  - `users` - 用户账号（角色：us_finance/cn_finance/admin）
  - `audit_logs` - 审计日志
- 状态枚举定义
- 索引策略（复合索引 + 部分索引）
- 外键约束和级联规则

**关键设计**:
- 支票号唯一性约束（deleted_at IS NULL）
- 乐观并发控制（row_version 字段）
- OCR 结果与业务记录解耦

### 3. 页面交互设计文档

**文件**: `收款系统_页面交互设计_V1.0.docx` / `.md`

**内容概要**:
- 交互决策汇总（8个关键决策点）
- Dashboard 首页（统计卡片 + 快捷入口）
- 支票采集页（4个状态：上传 → OCR识别 → 左图右表单 → 入库成功）
- 收款记录列表（Drawer 详情 + 状态变更）
- 扣款导入页（批量上传 + 逐一录入）
- 扣款异常列表（手动关联 Modal）
- 报表导出 & 客户主数据管理
- 公共 Blazor 组件清单（11个组件）
- 导航菜单结构（角色权限）

**关键交互**:
- OCR 置信度颜色规则：绿（≥0.85）/ 橙（0.60-0.85）/ 红（<0.60）
- Dashboard 数据刷新：定时轮询 30 秒
- 支票上传后自动切换为左图右表单布局
- 并发保护：携带 row_version，冲突时提示刷新

## 常见问题 (FAQ)

**Q: Word 和 Markdown 版本有什么区别？**
A: 内容完全一致，Word 版本保留原始格式和表格样式，Markdown 版本便于在代码编辑器中查看和搜索。

**Q: 如何快速了解项目？**
A: 建议按以下顺序阅读：
1. 技术栈架构决策（了解技术选型和整体架构）
2. 数据库设计（了解数据模型和表结构）
3. 页面交互设计（了解 UI/UX 和业务流程）

**Q: 设计文档是否会更新？**
A: 设计阶段文档已确认，可作为开发输入。如有调整会更新版本号（如 V1.1）。

**Q: 开发时应该参考哪个版本？**
A: 优先参考 Markdown 版本（便于搜索和复制），Word 版本作为备份参考。
