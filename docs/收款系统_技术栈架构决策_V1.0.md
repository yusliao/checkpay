| 1. 三大核心决策                                                   |
|                                                                       |
| Three Key Architecture Decisions                                      |
+-----------------------------------------------------------------------+

决策一：前后端不分离 → Blazor Server

  ----------------------------------------------------------------------------
  对比项React + API（分离）Blazor Server（不分离）✅
  ---------------- --------------------------- -------------------------------
  代码库           前端 + 后端，两套           一套 C# 代码库

  部署             两次部署，需配置跨域        一次部署，零跨域问题

  实时推送         需要额外集成 WebSocket      SignalR 内置，开箱即用

  类型安全         需手写 API DTO + 前端类型   共享同一份 C# 模型

  学习成本         需要会 TypeScript + React   只需会 C#

  OCR完成通知      前端轮询或额外WebSocket     服务端直接推送，自动刷新
  ----------------------------------------------------------------------------

+-----------------------------------------------------------------------+
| ✅ 结论：Blazor Server 是 .NET 10 全栈不分离的最佳实践选择。          |
|                                                                       |
| 一套代码、一次部署、C# 贯穿前后端，完全符合「运维简单」的核心诉求。   |
+-----------------------------------------------------------------------+

决策二：云平台 → Railway（替代 Azure App Service）

  ----------------------------------------------------------------------------------
  对比项Azure App ServiceRailway ✅
  --------------------- ----------------------------- ------------------------------
  月费（应用+数据库）   \~\$100/月                    \~\$10/月

  部署方式              需配置 Azure DevOps/Actions   连接 GitHub，push 自动部署

  数据库                需单独购买托管实例            内置 PostgreSQL，一键添加

  环境变量管理          App Configuration 服务        Web 界面直接设置

  适合规模              企业级，数百用户              小工具、内部系统 ✅

  学习成本              高（Azure 服务体系复杂）      极低，15分钟上手
  ----------------------------------------------------------------------------------

+-------------------------------------------------------------------------+
| ✅ 结论：两个用户的内部工具，Railway \$10/月完全够用，节省90%运维费用。 |
|                                                                         |
| 代码层面零改动，.NET 10 应用直接部署，Railway 自动识别构建。            |
+-------------------------------------------------------------------------+

决策三：OCR → Azure Document Intelligence（保留）

尽管应用层迁移到 Railway，OCR 服务仍使用 Azure Document Intelligence，原因：

  -----------------------------------------------------------------------------------------------
  原因说明
  ---------------------- ------------------------------------------------------------------------
  无可替代的预训练模型   prebuilt-check 专为美国银行支票训练，开箱识别支票号/金额/日期

  按调用量计费           不是包月，每次 \$0.01，两人使用量下费用极低

  Custom Model 支持      二期扫描件 OCR 可在同平台训练自定义模型，无需换方案

  与 Blob Storage 集成   支票图片存 Azure Blob，直接传 URL 给 Document Intelligence，零传输成本
  -----------------------------------------------------------------------------------------------

| 4. Solution 项目结构                                              |
|                                                                       |
| .NET 10 Solution Layout                                               |
+-----------------------------------------------------------------------+

遵循垂直切片（Vertical Slice）+ 整洁架构思想，单 Solution 包含以下项目：

  ----------------------------- ------------------ ---------------------------------------------
  项目类型职责

  `CheckPay.Web`              *Blazor Server*    UI 页面、组件、布局。所有 .razor 文件在此

  `CheckPay.Application`      *Class Library*    业务用例（Commands/Queries）、接口定义、DTO

  `CheckPay.Domain`           *Class Library*    领域模型（实体、枚举、业务规则、状态机）

  `CheckPay.Infrastructure`   *Class Library*    EF Core DbContext、仓储实现、Azure 客户端

  `CheckPay.Worker`           *Worker Service*   OCR 异步任务消费者，后台轮询处理队列

  `CheckPay.Tests`            *xUnit*            单元测试 + 集成测试（Testcontainers）
  ----------------------------- ------------------ ---------------------------------------------

4.1 依赖方向

+-----------------------------------------------------------------------+
| Web → Application → Domain （向内依赖，Domain 不依赖任何层）          |
|                                                                       |
| Infrastructure → Application （实现 Application 定义的接口）          |
|                                                                       |
| Worker → Application → Infrastructure（Worker 复用同一套业务逻辑）    |
+-----------------------------------------------------------------------+

5.1 初次部署步骤（一次性）

  -----------------------------------------------------------------------------------------------------------
  步骤操作说明
  ---------- ---------------------------------- -------------------------------------------------------------
  1          注册 Railway 账号                  railway.app，绑定 GitHub 账号

  2          New Project → Deploy from GitHub   选择代码仓库，Railway 自动识别 .NET 项目

  3          Add PostgreSQL Plugin              项目内一键添加，DATABASE_URL 自动注入

  4          设置环境变量                       AZURE_DOC_INTEL_KEY、AZURE_BLOB_CONN、ENTRA_* 等

  5          设置启动项目                       Railway → Settings → Start Command: dotnet CheckPay.Web.dll

  6          绑定自定义域名（可选）             Settings → Domains，Railway 自动申请 SSL
  -----------------------------------------------------------------------------------------------------------

+-----------------------------------------------------------------------+
| 6. OCR 异步处理架构                                               |
|                                                                       |
| OCR Worker Architecture                                               |
+-----------------------------------------------------------------------+

两人使用的规模，不需要 Azure Service Bus。用 .NET 内置的 Channel\<T\> 实现内存队列，简单可靠。

+-----------------------------------------------------------------------+
| 流程一（支票原件）：                                                  |
|                                                                       |
| 用户上传图片 → Blob Storage → 写入 OcrChannel → Worker 消费           |
|                                                                       |
| → 调用 prebuilt-check → 写入 ocr_results 表                           |
|                                                                       |
| → Blazor SignalR 推送通知 → 前端自动更新复核表单                      |
|                                                                       |
| 流程二（扫描件，一期）：                                              |
|                                                                       |
| 用户上传图片 → Blob Storage → 左右分屏界面 → 用户手动录入五个字段     |
|                                                                       |
| → 确认后写入 debit_records（无 OCR，直接入库）                        |
|                                                                       |
| 流程二（二期，Custom Model 训练完成后）：                             |
|                                                                       |
| 用户上传图片 → Blob Storage → 写入 OcrChannel → Worker 消费           |
|                                                                       |
| → 调用 custom-debit-v1 → 预填五个字段 → 用户复核确认                  |
+-----------------------------------------------------------------------+

6.1 Worker 与 Web 共存方式

  -------------------------------------------------------------------------------------------------------------------------------
  方式说明推荐
  -------------------------- ------------------------------------------------------------------------- --------------------------
  同进程（IHostedService）   Worker 作为 BackgroundService 注册在 Web 项目中，Railway 只部署一个服务   ✅ 一期推荐

  独立 Worker 进程           CheckPay.Worker 单独部署为 Railway 的第二个 Service                       二期考虑，现阶段过度设计
  -------------------------------------------------------------------------------------------------------------------------------

+-----------------------------------------------------------------------+
| ✅ 一期将 OcrWorker 注册为 Web 项目的 BackgroundService。             |
|                                                                       |
| Railway 只需部署一个服务，费用不变，运维更简单。                      |
|                                                                       |
| 二期 OCR 量增大时，再拆分为独立 Worker 服务，代码改动极小。           |
